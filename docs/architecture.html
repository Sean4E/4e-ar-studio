How the pieces fit together · What's fast vs slow · When to expect what

Where things live

Three hosting layers. Different speeds, different jobs — use the right one for the right thing.

🎨 Studio UI

GitHub Pages hosts the editor HTML, modules, CSS. You iterate the studio by pushing to the repo.

sean4e.github.io/4e-ar-studio/studio/

📦 Assets & library

GitHub Pages hosts every uploaded GLB, thumbnail, audio, video, image-target PNG, splash logo, PWA icon. Free, CDN-backed, cross-origin OK.

sean4e.github.io/4e-ar-studio/assets/{id}/…

🚀 Published player

Firebase Hosting serves only the player shell (tiny). Everything else (assets/ studio/ projects.html) is excluded from Firebase deploys.

ar-studio-4e.web.app/player-v2.html?id=…

🗂 Project data

Firestore stores one JSON document per project — objects, prefabs, targets, splash config, PWA URLs. Tiny payloads, fast reads.

firestore/ar_apps/{projectId}

⚙ CI

GitHub Actions deploys the player shell to Firebase. Path-scoped: triggers only when actual player files change.

.github/workflows/firebase-deploy.yml

🔑 Auth

Firebase service account held as a GitHub Actions secret. Lets the workflow deploy without interactive login.

FIREBASE_SERVICE_ACCOUNT_AR_STUDIO_4E

What propagates when — and how long to wait

Instant Studio edits — transforms, inspector changes, material toggles, hierarchy. All happen in RAM on your browser. No network, no wait. 0 ms
< 1 s Save to Firestore — serialised project JSON committed. Visible on other devices / published apps within seconds. ~200-800 ms
< 2 s Asset upload — GLB / image / video sent to GitHub via Contents API. Creates a commit on main. Returns a URL immediately. 1-3 s
Variable GitHub Pages propagation — the file exists in the repo but is not yet served by the CDN. This is the ONE real wait in the pipeline. Publish modal polls all asset URLs and shows live progress (Assets live: N / total) until everything is 200. 30-120 s
~45 s Firebase player deploy — only runs when you push changes to player-v2.html, config.js, sw.js, samples/, firebase.json, or the workflow itself. Asset uploads never trigger it. 30-60 s
Instant Published app launch — Firestore fetch + preload only this scene's own assets (GLBs, target PNGs, audio). Videos are background prefetch so splash never waits for big files. 1-3 s

The publish workflow (what the modal actually does)

1 Upload any pending files to GitHub: image targets → assets/{id}/targets/…, prefab GLBs → assets/{id}/prefabs/…, splash logo, PWA icons (regenerated from logo), splash-logo thumbnails. instant per file
2 Call Studio.saveProject() → serialise state, write to Firestore. < 1 s
3 Gather every github.io URL the published app will ever load. Start polling all of them in parallel with HEAD requests. < 1 s
4 Progress bar in modal — "Assets live: 5 / 12 · 18 s". Polls every 4 s. QR code is hidden until 100 %. 30-120 s
5 ✅ Live and ready to scan! Shows player version, QR code, theme-coloured so it matches the splash. Test URL + download QR available. instant

Speed tricks already in place

🎬 Splash on first paint Startup

Splash DOM ships in the player HTML. A sync head-script pulls cached colours/logo from localStorage before body paint. Repeat visits show the branded splash from frame 1 — no Firestore round-trip needed for visuals.

⚡ Asset preload split Cold start

Blocking preload = only what's in this scene (GLBs of instances, image-target luminance PNGs, audio). Videos fire-and-forget so splash doesn't wait for MBs. Library prefabs are never preloaded.

🧠 8th Wall chunk preload Cold start

data-preload-chunks="slam,face" on the engine-binary script tag warms up tracking algorithms during script load — not on first use.

🗂 Service worker PWA

Network-first with cache fallback for the player shell. Installed PWA launches get shell from cache instantly; online launches always fetch fresh.

🧵 URL dedupe Preload

The preloader uses a Set: five instances of one prefab = one fetch, not five. Same GLB shared across image targets = one download, many renders.

📸 Thumbnails rendered once Library

Offscreen WebGLRenderer auto-frames each prefab and saves a 128×128 PNG next to its GLB. Loaded from disk on subsequent library views — no runtime rendering cost.

🌐 Cross-origin CORS Fetch

GitHub Pages serves all static files with Access-Control-Allow-Origin: *. The player fetches GLBs / audio / videos / images cleanly from a different origin — no preflight.

🕒 No-cache on HTML/JS Fresh code

Firebase Hosting serves *.html, *.js, sw.js with no-cache. New deploys reach users on their next launch — no HTTP-cache TTL delay.

Rules of thumb

⚡
Studio changes ship almost instantly. Push to main → GitHub Pages rebuilds (1-2 min). Hard-refresh the studio tab to see new code. HTML is not HTTP-cached so a normal refresh usually suffices.
📦
Asset uploads go to GitHub, not Firebase. Every upload triggers a commit. Don't worry about Firebase storage — assets/ is in the firebase.json ignore list, each Firebase deploy stays tiny.
🎯
Publishing waits for all assets to propagate. The modal polls every GitHub-hosted URL. You see "Assets live: N / total" until it's 100 % — then the QR appears. No more guessing whether the app is really ready.
🚀
Player code updates auto-deploy to Firebase. When you push changes to player-v2.html, sw.js, config.js, firebase.json, or samples/, GitHub Actions runs FirebaseExtended/action-hosting-deploy. Studio-upload commits are path-scoped out of the trigger.
🔄
Version number is everywhere. PLAYER_VERSION is stamped on the splash branding line. If you don't see the expected version on a published app, hard-refresh once — browser HTML cache is the usual culprit (and it shouldn't happen after v3.10.2's no-cache headers).
🌀
Browser "empty state" before splash is unavoidable for non-PWA. When a QR scan opens a fresh browser tab, the browser shows its own blank page until your HTML bytes arrive. Only installed PWAs skip this (iOS uses theme-color meta for native splash — already set).

Anti-patterns — avoid these

✕ Don't host assets on Firebase

Firebase Hosting has a storage quota (10 GB Spark) that stacks across retained versions. Assets stay on Pages. firebase.json ignore list already enforces this.

✕ Don't trigger deploy on every commit

The workflow is path-scoped to player-shell files only. Studio asset-upload commits never trigger deploys. Keeps Firebase deploy count low → quota stays healthy.

✕ Don't inline assets into Firestore

Firestore docs have a 1 MB limit. We store URLs, not bytes. Thumbnails are PNGs on Pages; splash logos, PWA icons, GLBs, target imagery — all URLs.

✕ Don't preload what the scene doesn't use

Library prefabs that aren't instantiated are not preloaded by the player. We learned this the slow way — shipping a fix in v3.10.1 to cut cold start back to fast.

If you see…

🐌 Slow cold launch

Check which assets the scene uses + their total size. Big GLBs (> 10 MB) will dominate. Videos don't block — GLBs do.

🟦 404 on an asset right after upload

GitHub Pages propagation — normal for the first ~60 seconds after upload. Studio retries GLB loads; the publish modal waits until every asset is 200 before showing ✅.

🧊 Stuck on old code in browser

Shouldn't happen after v3.10.2's no-cache headers. If it does: hard-refresh once (Ctrl+Shift+R / Cmd+Shift+R). For installed PWAs, the service worker picks up new code on the next online launch.

🧨 Firebase 429 on deploy

Hosting storage quota. Open Firebase Console → Hosting → release retention. Set to a low number (2-5). The rare case — firebase.json's ignore list keeps releases tiny.
